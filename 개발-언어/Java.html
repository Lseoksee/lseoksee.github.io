    문자열 (String I/O)

    한 단어 또는 한 문자열 처리

    • 문자 하나하나 또는 문자열을 설정된 인코딩에 맞게 변환해서 문자열 다룸
      • Supplementary Planes 에 해당하는 문자열은 char[]를 하나 더 읽어야 됨
      • 확인법: Character.isSupplementaryCodePoint(codePoint)
    • 기본 Stream에 문자 인코딩 처리를 위한 인코더가 추가된 거라 당연하게도 바이너리를 해당 객체들로 다루면 데이터 깨짐
    • 자바에서 문자열을 저장하는 방식
      • 문자열을 UTF-16 으로 인코딩 하고 있다가 따로 참조할일 있을때 디코딩 해서 표현

    InputStreamReader & OutputStreamWriter

    • 이걸 String I/O로 변환 시켜 줌

    바이너리 (binry I/O)

    데이터를 1바이트 단위로 처리함

    • 해당 객체들로 텍스트 문자도 처리가 가능하지만 추가적인 인코딩 처리가 필요하고, 비효율적임
    • 인코딩 마다 각각 할당되는 바이트 수가 달라서 1바이트 읽고 그걸 문자열로 인코딩 한다면 당연하게도 문자가 깨짐
    • 그렇기에 모든 바이트 읽고 그걸 인코딩 해서 전체 바이트를 해석하게 해야 함

      • 중간에 바이트가 끊기기 떄문에 문자열이 깨짐

    Java 에서의 바이트 표현

    • 예를들어 0xFF 즉 1111 1111 은 10진수로 표현하면 255
      • unsigned byte 로 표현 할 수 있는 최대값
    • 하지만 byte 자료형은 부호가 있는 2의 보수로 표현되기에 -128 ~ 127 까지의 숫자만 저장 가능하다
    • unsigned byte 자료를 다룰땐 byte 로 형변환 하여 2의 보수법 으로 해석 하게한다
    • 실제로 File 이나 기타 바이너리 자료를 다룰 때 byte[] 단위로 저장될텐데 각 배열 값을 바이트 값을 2의 보수법으로 해석 하게 된다, 그래서 음수가 나올 수 있는것

    업케스팅 & 다운케스팅

    동적 바인딩

    • 부모로 부터 Overriding된 메소드는 부모 메소드는 무시되고 자식 메소드가 호출된다

    업케스팅 (Upcasting)

    • 자식 클래스를 부모 객체로 초기화 하는 방법
    • 이렇게 초기화 하더라도 동적바인딩이 작동하여 자식에 구현된 메소드만 호출한다

    사용하는 이유?

    • 가장 큰 의문이 그냥 자식 그 자체로 객체를 초기화 하더라도 동적 바인딩 같은 동작에는 아무런 문제가 없다.

    • 그래서 업케스팅이 필요한 이유를 가장 잘 설명한 게 아래 예제 이다

      • Dog 와 Cat 둘 다 Animal 상속 받은 상태라 내가 설계하는 메소드는 Cat, Dog 와 같이 Animal을 상속받은 모든 객체면 상관없는 상황이라고 가정한다
      • 그럴때는 위 makeAnimalSound() 메소드 처럼 Animal 이라는 타입으로 뭉뚱그려서 타입을 명시 할 수 있기 때문이다.
      • 그래서 업케스팅은 그냥 하위 객체 전체의 타입을 명시하기 위한 수단 정도로만 기억하자

    다운 케스팅 (DownCasting)

    • 위 예제 처럼 makeAnimalSound() 에서 Animal 타입에 공통 함수인 sound()를 수행하고
    • 각각의 하위 클래스에 구현된 고유한 메소드나, 변수를 사용하기 위해 완전히 자식 클래스로 전환 하는 것
    • 저렇게 뭉뚱그려서 Animal 로 타입을 지정한 경우 호출할때 사용되는 객체 각각에 대해 따로 처리할 때 사용한다
